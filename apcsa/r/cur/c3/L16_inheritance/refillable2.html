It doesn't make much sense to set up a CokeCooler class that provides exactly the functionality of Refillable—in that case, we should just use Refillable as is. But suppose we want to provide some CokeCooler-specific output, by printing the amount of drinks in the cooler during each refill or useUp call. We do this by providing our own refill and useUp methods—overriding those of Refillable. We can call the corresponding superclasses overridden method using the super keyword; for example, super.refill(amount) and super.useUp(amount) access the overridden methods passing amount as the argument. Here is the code. 

public class CokeCooler extends Refillable {

   public CokeCooler (int initialAmount) {
      super (initialAmount);
   }

   public void useUp (int amount) {
      super.useUp (amount);
      System.out.println ("There are now " +
	    currentAmount ( ) + " Cokes in the cooler.");
   }

   public void refill (int amount) {
      super.refill (amount);
      System.out.println ("There are now " +
	    currentAmount ( ) + " Cokes in the cooler.");
   }
}

CokeCooler inherits Refillable's currentAmount method, so it can use it without needing an explicit object reference of type Refillable. Each of the methods listed above first calls its Refillable counterpart, then (in useUp and refill) prints out the CokeCooler specific function of printing the current amount to the console.

In fact, the first thing every subclass constructor must do is call the superclass constructor; if the super call is omitted, Java substitutes a call to the 0-argument superclass constructor to try to ensure that all the object's instance variables are properly initialized.
