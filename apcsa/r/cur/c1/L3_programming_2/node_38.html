This lesson involved further work with classes, specifically the Refillable and Name classes. Issues arising from Refillable included the following:

  • Informative naming of methods and instance variables. We will use nouns to name typed methods and imperative verbs for void methods.
  • Adding functionality to the class. To compute the average of values given as arguments to useUp, we considered first what extra state was needed to provide the new operation, then what changes to the existing methods had to be made.
  • Testing. Alongside the Refillable class, we provided a "tester" class that calls all the Refillable method, printing results to the BlueJ console via System.out.println. Tester classes help to simplify the somewhat tedious process of producing evidence that the Refillable code works as specified.

The Name class illustrated more features of typical classes: local variables that are used to keep track of intermediate results in a computation, private "helper" methods that help other methods do their jobs. We considered in some detail the design of the helper method, first providing three helpers (one for each piece of the name), then combining them, using a parameter, into a single more general method.

The Name class also made heavy use of String objects. The String class is supplied in Java; it represents a sequence of characters that you can type at the keyboard. String constants are surrounded by double-quote characters. Characters in a String are numbered starting at 0. Two String objects can be concatenated—"glued" together— to form a new String using the "+" operator. (Thus the "+" operator performs different in different contexts.) A String can be empty, that is, it can contain 0 characters. Blanks can be part of a String; a String that contains only one or more blanks and nothing else is not empty, since the blanks contribute to its length.

There are 60 methods in the String class. The Name class uses four:

  • substring, which returns a part of the String;
  • toLowerCase, which returns the conversion of lower-case letters in the String to upper case;
  • toUpperCase, which returns the conversion of upper-case letters in the String to lower case;
  • trim, which returns the result of removing blank characters from the ends of the String.

All of these methods return a newly constructed String object; the object used to call the methods isn't changed. Two versions of substring are provided in Java, a version with one argument and another with two. Java knows which one to call because the signature (the list of parameters) is different.

The design of the helper method for the Name constructor revealed an important property of method calling. Namely, a method is given a copy of each of its arguments. This preserves the argument values in the calling code. For String objects, however—and for that matter, any kind of object—a reference to the object is what gets copied. We described the call to the Name constructor in some detail, using boxes and arrows in diagrams to portray reference values.

The modification to the Refillable class led us to a new data type, named double, whose values all have a fractional (digits following the decimal point). Such values are stored in a form similar to scientific notation, that is, as some value times some power of 10. Since when we move the decimal point 1 place to the left and add 1 to the power of 10, we get the same value, this format is called floating point.

Finally, we noticed that big enough values cause undetected errors. That's because an int is represented in a limited amount of memory, namely 32 "bits" (binary digits). That means we can only represent 232 actual values using an int.
