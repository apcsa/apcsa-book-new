In the first section of this lesson, we examined one of the first steps that a programmer takes before creating a program: understanding the objects that s/he will write about. There are two important features of objects: behavior and state.

  • Behaviors are actions that an object is able to take. Each behavior has a name that should describe its action. For the alarm clock, an example was the behavior sound the alarm.
  • State is composed of variables, each of which is a piece of information known to the programming object. Each variable has a name—what the information is about—and a value—the piece of information itself. For the alarm clock, an example was named alarm time, which could take a value like 8:15 am.

Behaviors and state together provide a way of describing an object. We saw that a description based only on behaviors can be clumsy. A large set of behaviors can often be generalized, using information that's part of the object's state or information that's behavior-specific. The latter kind of information is referred to as a parameter. We explored these concepts in several everyday objects (alarm clocks, cell phones, toasters, and MP3 players).

Next, we encountered the idea of a class—a sort of blueprint or pattern from which objects with a given behavior and state may be produced.  Each object produced from a class is an instance of the class.

We examined several examples of classes and instances: shoes, online restaurant reservation services, sewing patterns and clothes made from them, songs, restaurants, books, and playing cards.

We will soon see that every Java program is a collection of classes. When run, the program will produce objects that will interact, produce output, and so on.

Finally, we encountered objects in Java programs. Objects have variables, and the state of these variables will be represented using numeric values, text values, Boolean (yes-or-no) values, and combinations of these three. Objects also have methods that represent their behaviors.

In addition, we took an extended look into a program that draws a game board, specifically at how its methods are called in a sequence.
