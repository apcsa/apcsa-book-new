Our programs so far have involved three kinds of statements:

  • assignment statements;
  • method calls;
  • return statements.

When we call a method from the Object Bench, we know exactly what statements are executed as a result, and exactly what sequence they will be executed. For example, calling the constructor of the Refillable class always executes its three assignment statements in the same order: 

   myAmount = initialAmount;  // always executed first
   myUseUpCallCount = 0;      // always executed second
   myTotalUse = 0;            // always executed third

We haven't yet seen a way to sometimes not execute a statement, even though it might not make any sense to execute. In the original version of the Refillable class, we were not able to validate arguments by handling nonsense values or values that don't make sense in the context of the program differently from reasonable values.

We'll use conditional execution to handle this. It'll also be useful for designing flexible code that can handle a variety of reasonable situations.
