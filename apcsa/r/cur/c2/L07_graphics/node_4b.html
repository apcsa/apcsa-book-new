Let's look at the whole class again, focusing more closely at the paintComponent method: 

public class Greetings extends JPanel {
   private int myWindowWidth = 300;
   private int myWindowHeight = 400;
	
   public Greetings() {
      JFrame easel = new JFrame();		
      easel.setSize (myWindowWidth, myWindowHeight);
      easel.setDefaultCloseOperation (JFrame.EXIT_ON_CLOSE);
      easel.add (this);
      easel.setVisible (true);
   }
	
   public void paintComponent (Graphics g) {
      g.drawString ("Hi there!", 150, 200);	
   }
}

The paintComponent method is what does the actual drawing. The drawString method will draw a string on a JPanel (the canvas). But we are getting ahead of ourself: where in this class is the paintComponent method ever getting called?

Look closely. You'll notice that nothing in the Greetings class calls paintComponent. As you might have guessed, the JPanel class, which the Greetings class is acting like, is doing something sneaky.

It is called whenever the underlying system—that is, Java—thinks the drawing window needs to be updated, for example, immediately after the call to setVisible, or after the window is resized, or after an overlapping window is moved or deleted. (We saw a similar situation in lesson 1, where the "tick" method in the alarm clock was called by some clock internal process.)

If the JFrame was never made visible, or if the Greetings object was never attached to the JFrame, then paintComponent would never be called.
